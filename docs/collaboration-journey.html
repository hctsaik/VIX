Human + AI · System Design Case Study

VIX 開發紀實:
如何與 AI 協作打造一套系統

一個真實案例 —— 從「我訓練 YOLO/DINO 卻不知道資料加了有沒有用」的模糊困惑,到一份四方達成共識的可落地架構。記錄人與 AI 一起把問題想清楚、再想對的全過程。

01緣起:一個說不清楚的困惑

使用者並沒有帶著明確規格來。他帶來的是一種「訓練現場的焦慮」。

「我在訓練 DINO、YOLO 的時候,其實在可解釋性、以及我增加了什麼資料進去會讓模型變得更好、或是讓定義變不一致,我並不知道,這也很困擾我。」

他同時附上了一份初步的系統草案(VIX),設計是:批次掃描本機資料夾、用 YOLO 信心度 + DINOv2 特徵距離篩出可疑影像、用 Streamlit 儀表板人工覆核、確認後把實體檔案搬移到不同資料夾(pass / review / golden)。

這一步的關鍵動作

沒有立刻照著草案寫程式。AI 先做的是:判斷使用者「真正的問題」和「他以為的問題」之間有沒有落差。這個落差,往往才是專案成敗的分水嶺。

02第一個轉折:AI 先幫你把問題重構

使用者把它叫「可解釋性 (eXplainability)」。但真正的痛,根本不是傳統 XAI。

使用者原本的框架

「我需要可解釋性。」
(腦中想的是 Grad-CAM、attention map ——「為什麼模型把這張預測成 bubble?」)

AI 重構後的框架

真正的痛是兩個資料層面的問題,屬於 Data-Centric AI,跟模型可解釋性無關。

真正的痛點它的本質對應的 SOTA 手段
加了哪批資料讓模型變好/變壞?資料歸因 (Data Attribution)凍結測試集 + 版控 + ablation(地基)→ Dataset Cartography → TRAK(進階)
加的資料讓 bubble/reflection 定義漂掉?標註品質 + 概念漂移cleanlab/label-error + 凍結參照漂移監控 + 多人一致性 κ

AI 同時點出草案裡幾個會踩雷的技術決定,並給了修正方向:

原草案問題修正
UMAP 2D 距離拿來做篩選判定UMAP 會扭曲全域距離UMAP 只用於「看」,判定在原始 embedding 空間做
centroid + cosine 距離假設類別是球形、單峰改 kNN(或 Mahalanobis)
固定 0.85 信心門檻未校準、拍腦袋per-class 校準門檻 → Conformal Prediction
用實體資料夾搬移表達狀態競態、難稽核(在 FiftyOne 中是反模式)用資料庫 tags/fields + view
假設 golden set 是乾淨的它幾乎從不乾淨定期用 cleanlab 自審 golden set

→ 還提醒:閉環「review → golden → 重訓 → 再篩」會放大模型自身偏差(定義被模型而非真實世界主導),需要凍結參照點來防。

03協作模式:為什麼用「多代理人辯論」

使用者下了一個關鍵指令:用 multi-agent 不斷思考與討論,直到達成共識,並全程記錄。

為什麼是多代理人?

單一視角會有盲點:演算法專家容易過度工程,產品經理容易砍掉必要的嚴謹。讓不同立場的 AI 角色彼此對立、互相挑戰,能逼出「沒有人能單獨想到」的折衷。這是刻意製造「建設性衝突」。

為什麼基於 FiftyOne?

使用者點名 voxel51/fiftyone。它已經做掉 80% 的輪子:embedding 視覺化、相似度/離群、label-error、標註整合。站在成熟工具上,而不是重造它。AI 全程上官網查證 API,而非憑記憶臆測。

角色 A

演算法

embedding、距離度量、label-error、cartography、門檻、漂移、資料歸因

角色 B

系統架構

build-vs-reuse 邊界、資料模型、UI、部署拓樸

角色 C

MLOps / 治理

凍結評估集、版控、閉環安全、漂移治理、可重現 ablation

角色 D

產品 / MVP

分期、最小可用範圍、刪減清單、效益與採用

04多代理人歷程:發散 → 收斂 → 共識

兩輪、八次代理人發言,全程寫成 Markdown,可追溯、可交接。

ROUND 1 · 發散 (Divergent)

四個角色在不知道彼此結論下,各自獨立提案

攤開所有觀點 —— 也攤開所有分歧。共識點很快浮現(基於 FiftyOne、UMAP 只供顯示、OSS 無原生版控需用 DVC),但也出現 7 個明確衝突,核心是一條張力:

產品 D:「其他角色合計超過 3 個月工程量,但使用者今天的痛點可以用 2 週解決 80%。過度工程的最大風險不是技術債,是這個工具根本沒人用完就被放棄。」
ROUND 2 · 收斂 (Convergent)

用「分期」當作調和總框架,逐條對撞 7 個衝突

關鍵頓悟:大多數衝突不是「誰對誰錯」,而是「該放在哪一期」。把產品 D 的分期框架當骨架,把演算法/架構/治理的高級項目分配到正確階段,衝突就化解了。

衝突化解方式
廢除實體資料夾 vs 不破壞既有腳本FiftyOne 當唯一事實來源 + 單向 export 轉接器把結果匯回既有資料夾(零改動)
kNN vs cosine 距離偽衝突 —— compute_similarity 的 kNN 不比 centroid 貴,直接採 kNN
嚴謹治理 vs 單人作業官僚化guardrail 依團隊規模分期:單人只留 3 項便宜防護,雙簽延到多人時
CONSENSUS · 達成

四角色全部「有條件是」,9 條件彼此相容

主持人裁決了唯一的張力(reference model 在 v0.1 = frozen pretrained DINOv2,不另訓練;完整版任務模型延 v0.2),讓所有條件同時成立。共識鎖定。

05共識成果:一份可開工的分期架構

核心結論 —— VIX 是建立在 FiftyOne 上的 Data-Centric AI 資料守門員,用分期控制複雜度,用凍結參照點防止閉環放大模型偏差。

v0.1 —「可見性」~2–3 週 · 單人本機 · 無 API

  • 既有資料夾 → 單一 FiftyOne Dataset(tags/fields 為狀態,不搬檔)
  • DINOv2 embedding + compute_similarity kNN 離群分數 + YOLO 信心 → tag
  • FiftyOne App 原生 UI 覆核(Lasso 選點 + tagging,不自建 plugin)
  • 三項便宜防護:凍結 golden test set + frozen anchor + frozen DINOv2 reference 打分(超閾值→單人自我 gate + 書面自審)
  • 單向 export 轉接器:golden → 資料夾 + YOLO txt + data.yaml(既有訓練腳本零改動)

v0.2 —「可追溯」~+4–6 週

  • DVC + W&B + 完整 ablation 協定(系統化回答「這批資料有沒有用」)
  • 門檻升級 per-class calibrated(temperature scaling)· cleanlab 橋接 · 漂移 KS test
  • FastAPI 觸發層接 CI/訓練 pipeline

v1.0 —「可制度化」~+8 週

  • 自建 FiftyOne Operator/Panel(覆核按鈕、逐張卡)· Dataset Cartography · Conformal Prediction
  • 多標註者雙簽 + 每季 re-audit + Cohen's κ 定義評審 · Mahalanobis 補強

完整規格、Build-vs-Reuse 對照、資料模型 schema、9 條 binding 條件、v0.1 交付清單,見 discussion/consensus.md。

06一個追問:當你挑戰 AI 的根基假設

共識鎖定後,使用者沒有照單全收,而是反問:「我需要直接用 FiftyOne,還是能自己開發類似功能?優缺點是什麼?」—— 這一問直接質疑了整份共識的地基。

好的協作姿勢,是把這種追問視為健康訊號。AI 沒有辯護先前的結論,而是再開一場三方辯論(正方直接用 / 反方自建 / 裁判決策框架),並全程上官網查證授權與部署能力。

⚠ 關鍵時刻:三個 AI 代理人「說法打架」

關於 fiftyone-brain 套件的授權,代理人給出互相矛盾的答案:

代理人說法
正方 / 反方Apache 2.0(開源、商用自由)
裁判閉源 freeware,禁反編譯/轉賣

授權是「能不能商用、要不要綁定」的決勝因素。AI 沒有挑一個聽起來合理的版本寫進文件,而是親自抓 PyPI metadata 與 GitHub LICENSE 原文驗證 —— 裁定:兩者目前皆為 Apache 2.0。裁判錯了(那是 brain 套件早期版本的舊狀況,後來已重新授權)。

辯論的真正共識:爭的不是「用不用」,而是「業務邏輯該不該綁進去」

直接委託給 FiftyOne

embedding 計算、相似度/kNN、App 互動式視覺化 + Lasso、標註整合 —— 這些自建要 3–6 個月,且真正難的部分不值得重造。

必須是 VIX 自己的(可獨立存活)

離群分數定義、門檻策略、凍結參照、決策紀錄、匯出邏輯 —— 寫成獨立 module,只透過一層薄 FiftyOneAdapter 接觸 FiftyOne。

結論:直接用 FiftyOne,但當「函式庫」用,並用一層薄 adapter 把領域邏輯解耦。既不浪費時間重造輪子,也不把自己鎖死 —— 日後要抽換 FiftyOne,只改一個檔案。何時才該改自建?資料不得外流且需 air-gap、UI 必須深嵌既有 MES、或需求極簡到用不上它 9 成功能時。詳見 round-03-use-vs-build.md。

再追問:保密、不能外流、又沒預算 —— 還能用嗎?

使用者再丟出三個真實世界的硬約束。AI 又查了一輪官方文件,結論可能跟直覺相反:

「資料不能外流」並不等於「必須自建」。FiftyOne OSS 100% 在本機/離線跑(Python + 本機 MongoDB + localhost App),預設組態下影像與標註沒有離開機器的路徑。

使用者的疑問查證後的答案
有完整原始碼嗎?有。core 與 brain 皆 Apache 2.0,且 GitHub repo 含完整可稽核實作(非空殼,親查 similarity.py 62KB 等)→ 可讀、可改、可 fork
有費用嗎?$0。Apache 2.0 對已發佈版本不可撤回 → 永久可用,絕不會被鎖死
後顧之憂?三個watch-item:① MongoDB 是 SSPL(內部用無妨,別當 SaaS 對外賣);② 無 RBAC/不可竄改審計(用 DecisionLog append-only + OS 層權限補);③ 法規若強制 SSO/法庭級審計才需另投資

這三個約束沒有推翻「直接用 FiftyOne」的結論,反而強化了 adapter 解耦 —— 合規邏輯留在 VIX 自己手裡,FiftyOne 只當地端的視覺化引擎。詳見 round-04-airgap-licensing.md。

07給同事的方法論:如何與 AI 協作開發系統

這份案例真正想傳達的,不是 VIX 本身,而是這套協作姿勢。把它當成可複製的 SOP。

1

先讓 AI 重構問題,別急著要程式碼

使用者開口要「可解釋性」,AI 卻先指出真正的問題是「資料歸因 + 概念漂移」。「我以為的需求」和「真正的問題」之間的落差,才是專案最該先處理的。

2

用持續性目標讓 AI 不中途停下

使用者用 /goal 設下「討論到共識為止」的條件,AI 就會自主跑完發散→收斂→共識的多輪迭代,而不是丟一個答案就停。把「期望的結束狀態」說清楚,比逐步下指令有效。

3

刻意製造「建設性的對立」

讓 AI 扮演立場相反的角色互相挑戰(嚴謹派 vs 務實派),逼出單一視角看不到的折衷。AI 最大的價值之一,是能同時、公正地扮演多個利害關係人。

4

站在成熟工具上,並要求 AI 查證而非臆測

基於 FiftyOne 而非從零造輪子;且要求 AI 上官方文件查證 API 能力(例:確認 OSS 沒有原生版本控制)。「先查證再下結論」是把 AI 從『聽起來合理』推向『真的可行』的關鍵紀律。

5

用「分期」調和理想與現實

多數架構爭論不是對錯,而是「該放哪一期」。分期讓「理想架構」與「2 週能交付」同時成立,直接化解過度工程的爭執。問「這要放 v0.1 還是 v1.0?」,而不是「要不要做?」

6

每一步都留下可追溯的文件

每輪的方向、重點、決策、共識、待澄清都寫成 Markdown。過程本身就是交接物——新成員讀 round-01 → round-02 → consensus,就能完整重建「為什麼這樣決定」。

7

人保留最終決策權,AI 提假設、列開放問題

遇到不確定(GPU?團隊人數?資料格式?),AI 採取明確的「工作假設」往前走,同時把問題列出來讓人覆寫,而不是卡住等答案。AI 負責推進與選項,人負責拍板。

8

讓 AI 查核「AI 自己的說法」,以一手來源為準

三個代理人對授權條款各執一詞時,正確動作不是投票多數決,而是回到 PyPI 與 GitHub LICENSE 原文驗證。AI 會自信地說錯話;把關鍵事實釘回一手來源,是讓「聽起來合理」變成「真的正確」的最後一道防線。

與 AI 協作開發系統,不是「叫它寫程式」。而是把它當成一個能重構問題、扮演多方、查證事實、推進決策、並完整留痕的協作夥伴 —— 人始終握著方向盤。
